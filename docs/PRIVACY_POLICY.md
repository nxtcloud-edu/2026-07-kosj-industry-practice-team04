# 「모아」 MVP 개인정보·위치정보 보관 및 삭제 정책

이 문서는 신고자 정보 보호 요구사항(SER-003, Issue #34)의 MVP 운영 기준이다. 실제 시민 데이터를 받기 전에 배포 환경의 저장소와 삭제 작업이 이 기준을 구현하는지 다시 검증한다.

## 1. 최소 수집 원칙

- 신고에는 사진과 위치가 필요하며, 위치정보 수집 동의를 받은 경우에만 접수한다.
- 연락처는 처리 결과 알림을 희망해 요청 본문에 직접 입력한 경우에만 선택 수집한다.
- 알림을 희망하지 않은 신고에는 연락처 필드를 만들거나 기본값으로 저장하지 않는다.
- 조회 토큰은 개인정보를 포함하지 않는 128비트 난수로 발급하고, 원문은 접수 응답에서 한 번만 전달한다. 저장소에는 SHA-256 해시만 보관한다.

## 2. 데이터별 보관 기간

| 데이터 | 수집 목적 | 보관 기간 | 기간 종료 후 처리 |
|---|---|---|---|
| 연락처(선택) | 처리 결과 알림 | 처리 완료 통보 후 30일 | 복구할 수 없도록 삭제 |
| 신고 사진 | 현장 확인·처리·재확인 | 처리 완료 후 6개월 | 삭제하거나 개인을 식별할 수 없도록 비식별화 |
| 신고 위치 | 담당 부서 배정·현장 확인·유사 신고 판정 | 처리 완료 후 6개월 | 삭제하거나 통계용 범위 정보로 비식별화 |
| 조회 토큰 해시 | 무로그인 상태 조회 권한 확인 | 사진·위치 보관 기간과 동일 | 삭제하여 조회 링크 무효화 |

사진을 학습 데이터로 사용하는 것은 이 보관 정책과 별개이며, 별도 동의를 받은 건만 비식별 처리 후 사용할 수 있다.

## 3. 조회와 응답 제한

- 상태 조회에는 접수번호와 올바른 조회 토큰이 모두 필요하다.
- 없는 접수번호와 잘못된 토큰은 동일한 `403` 응답으로 처리해 접수번호 존재 여부를 노출하지 않는다.
- 시민 상태 조회 응답에는 접수번호·상태·접수 시각만 포함한다. 사진·정확한 위치·연락처·토큰 또는 토큰 해시는 반환하지 않는다.
- 상태 조회 응답은 `Cache-Control: no-store`, `Referrer-Policy: no-referrer`를 사용한다.
- 조회 토큰, 연락처, 원본 위치와 전체 presigned URL을 로그에 남기지 않는다.

## 4. 삭제 실행 기준 — 구현됨 (Issue #59)

`backend/src/retention.js`의 예약 스윕이 이 기준을 실행한다. 서버 기동 시 1회 + 주기 실행(기본 1시간, `MOA_RETENTION_SWEEP_MS`).

1. 대표 문제가 `완료`로 전환된 시각(`completedAt`)을 기산점으로 삼는다. 완료를 되돌리면 기산도 초기화된다.
2. 완료 후 **30일**: 연락처 삭제. 완료 후 **6개월**: 사진 파일·사진 URL·주소·조회 토큰 해시 삭제, 좌표는 약 1km 격자로 비식별화.
3. 삭제 로그는 민감정보 원문 없이 건수만 남긴다.
4. 데이터 스냅숏 파일(`MOA_DATA_FILE`)에도 스윕 결과가 그대로 반영되므로 백업(스냅숏)과 본 저장소의 만료 기준이 일치한다.

검증: `backend/test/retention.test.mjs` — 30일 전 유지 / 30일 후 연락처만 / 6개월 후 사진·위치·토큰 삭제(업로드 파일 삭제 포함) / 미완료 신고 제외 / 완료 되돌림 기산 초기화. 운영용 외부 저장소(DynamoDB 등) 도입 시 같은 기준으로 재검증한다.
